# 二阶 / 三阶魔方拍照最短解应用

这是一个本地运行的魔方应用：选择二阶或三阶，上传六个面的照片，确认识别出的色块，然后按 HTM 度量输出严格最短复原步骤。

## 运行

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -r requirements.txt
python server.py
```

建议使用 Python 3.10 或更高版本。OpenCV 用于后端照片定位；如果未安装，应用仍可启动，但只会使用浏览器内置检测器。

打开终端显示的本地地址，例如：

```text
http://127.0.0.1:8765/
```

### 二阶识别与最短解

页面顶部可切换到“二阶（2×2）”。二阶没有中心块，应用会先把 24 个采样色块全局聚成 6 组（每组严格 4 块），再用 8 个角块的颜色组合和朝向约束自动确定颜色标签。若聚类结果无法组成合法角块，页面会提示校正照片方向或手动修改色块。

二阶求解器只建模角块排列与扭转，并把魔方整体的 24 种空间朝向视为同一个复原状态。它使用角块排列/朝向剪枝表和逐层 IDA*；只有所有更短深度都搜索失败后才返回当前解，因此输出是二阶通常定义下的 HTM 严格最少步数，最大搜索深度为 11。首次二阶求解会在内存中构建剪枝表，之后可直接复用。

### 原生最短解核心

高深度最短性证明由 C++20 核心执行。首次使用前编译程序并生成默认剪枝表：

```powershell
.\native\build.ps1
.\native\build_tables.ps1
```

默认会在 `.cache/native/` 生成约 67 MiB 的 16 对称联合表、约 42 MiB 的完整角块表，以及约 256 MiB 的深度 6 反向表。所有表均以内存映射方式只读加载并带完整数据校验；文件缺失或校验失败时服务端会安全回退到 Python 求解器。

项目也实现了 8 组完整六棱块模式库。三轴联合表启用后，它们在当前基准上会受随机内存访问限制，默认不加载；需要实验时可执行：

```powershell
.\native\build_tables.ps1 -IncludeEdgePdbs
$env:CUBE_NATIVE_EDGE_PDBS = "1"
python server.py
```

## 拍照识别

上传图片不要求是正方形，也不要求魔方面铺满整个画面。应用会通过 OpenCV 综合色块轮廓、图像边缘和 3x3 网格结构定位魔方面，估计四个外角，再进行透视矫正和九宫格取色。若 OpenCV 不可用，会自动降级到浏览器内置检测器。

色块取色会排除高亮、低饱和的镜面反光像素。普通块会同时观察中心与边缘，避免整片高光遮住真实底色；中心块采用环形区域取样，避免白色中心的蓝色 Logo 污染颜色原型。六面上传完成后，应用会对 48 个非中心块进行全局颜色分配，保证每种中心色最终恰好有 9 块。

每个已上传的面右上角会显示识别置信度。显示“请校正区域”或裁剪位置不准确时，点击 `⌗`，在原图上拖动四个角并应用。复杂背景、严重反光、遮挡或魔方过小时，建议直接使用四角校正。

## 六面约定

页面按 `U R F D L B` 六个位置上传图片：

- `U`：上面
- `R`：右面
- `F`：前面，也是输出解法的基准前面
- `D`：下面
- `L`：左面
- `B`：后面

上传后可以旋转每个面的 3x3 色块，也可以点击任意非中心色块手动改色。中心块固定为该面标签。

每张照片都应从该面的外侧正对拍摄，并按下面的相邻边朝上放置：

- `F`、`R`、`L`、`B`：`U` 边朝上；
- `U`：`B` 边朝上（也就是 `F` 边朝下）；
- `D`：`F` 边朝上。

如果拍摄时方向不同，请使用每个面右上角的旋转按钮校正。仅凭六张独立照片可能存在多个空间方向解释，应用不会在有歧义时静默猜测真实方向。

若想以另一个面作为前面基准，把该面的图片放到 `F` 位置，并按真实魔方的相邻关系重新放好其他五面。

## HTM 最短解

在 HTM 度量中，`R`、`R'` 和 `R2` 都计为 1 步。点击求解后，后端先用两阶段搜索返回一个可立即执行的快速解，再创建后台任务验证严格最短解。快速解会明确标记为“尚未证明最短”，不会冒充最短解。

后台验证优先使用 C++ 逐深度搜索和可采纳剪枝表。页面每秒轮询一次任务状态，证明完成后会自动用严格最短解替换快速解。

原生核心包含 88,179,840 状态的完整角块模式库、140,908,410 项的角扭转/棱翻转/UD 切片联合表、16 对称压缩、三轴下界、深度 6 反向状态表和紧凑转置表。任务会按当前队列负载动态继续分割，而不是只并行第一步；各深度仍按从小到大的顺序完整搜索，因此不会牺牲“严格最短”保证。

前台两阶段算法先快速产生可执行上界，并在交互预算内继续枚举第一阶段终点以收紧上界；后台 IDA* 只搜索更短深度。超时只表示证明尚未完成，不会把候选解错误标记为最短。

随机极难状态的严格最短搜索可能耗时较久，页面右侧可以调整“最短验证超时”。验证超时不再清空已经生成的快速解，页面会保留可用解法并说明其最短性尚未证明。

## 验证

```powershell
node tests\recognition.test.js
node tests\color.test.js
python -m unittest discover -s tests -v
python -m compileall cube_app server.py
```

实拍回归测试按文件名末尾的组号读取图片，例如第一组为 `initial/U1.jpg`、`R1.jpg`、`F1.jpg`、`D1.jpg`、`L1.jpg`、`B1.jpg`，第二组使用数字 `2`。未提供一组完整图片时，`test_real_images.py` 会自动跳过该组。

```powershell
python tests\extract_real_patches.py --group 1 | node tests\real_color.test.js --group 1
python tests\extract_real_patches.py --group 2 | node tests\real_color.test.js --group 2
```

可单独运行固定状态性能基准：

```powershell
python tests\benchmark_solver.py --timeout 30
```
